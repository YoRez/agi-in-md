## Structure First: Level 9 Diagnostic

---

### THE INITIAL CLAIM

**The `partial is None` guard in `Router.app` is a dispatch-optimization guard that has been silently reused as a method-collection mechanism. It correctly implements "keep the first fallback handler for dispatch" while incorrectly implying that the first partial-matching route's method set is sufficient for the RFC 7231 `Allow` header. The guard structurally prevents accumulation of method information across all routes matching the same path, making every multi-route REST resource produce a non-compliant 405.**

Falsifiable test: register two routes sharing a path:

```python
Router(routes=[
    Route("/data", get_handler, methods=["GET"]),
    Route("/data", post_handler, methods=["POST"]),
])
```

Send `DELETE /data`.

- `Route("/data", get_handler)` → path matches, method wrong → `Match.PARTIAL`. Guard: `partial is None` → True. `partial = Route_A`.
- `Route("/data", post_handler)` → path matches, method wrong → `Match.PARTIAL`. Guard: `partial is None` → **False. Route B's `POST` is discarded.**
- Response: `405 Allow: GET, HEAD` — `POST` is absent.
- RFC 7231 §7.4.1 requires: `Allow: GET, HEAD, POST`.

The guard on line `elif match == Match.PARTIAL and partial is None` is the exact site where compliance fails.

---

### THE THREE-EXPERT DIALECTIC

**Expert 1 — Defender:** The claim correctly locates the failure to a single guard and demonstrates the consequence. The variable `partial` (singular noun, set once) encodes the dispatch model: "one fallback handler." A correct method-collection model would need a set or list, updated on every PARTIAL match. The guard enforces the wrong data structure for the reporting requirement while being exactly right for the dispatch requirement. Both requirements live in the same loop; neither the variable name nor the code shape distinguishes them.

**Expert 2 — Attacker:** The claim targets the wrong fix. Even if you remove the `partial is None` guard and accumulate all partial-matching routes, the 405 response is generated inside `Route.handle`:

```python
async def handle(self, scope, receive, send):
    if self.methods and scope["method"] not in self.methods:
        headers = {"Allow": ", ".join(self.methods)}  # ← self.methods only
```

`Route.handle` recomputes the method check from its own `self.methods`, independently of whatever the router accumulated. `Router.app` calls `await partial.handle(scope, receive, send)` and then exits. The `handle` method has no parameter for an externally-computed allowed-method set. Fix the guard, and you've changed the data collected by the router while leaving unchanged the data used by the 405 generator. The fix requires changing `BaseRoute.handle`'s interface or moving 405 generation to the router — not changing the collection loop.

**Expert 3 — Probing:** You're both assuming 405 should be generated by routing infrastructure at all. But RFC 7231 says the `Allow` header must reflect methods "currently supported by the target resource." Resource-level method support is application logic: it depends on authorization policy, resource existence, feature flags — none of which the router knows. You both accept that route registration is a complete definition of method support. What happens when `GET /data` is registered but the authenticated user doesn't have GET access? The router claims `Allow: GET, HEAD` in a 405 for a DELETE from that user — which is false for that user. The router's method model is structurally incapable of expressing conditional method support. Should it be generating 405s at all?

### THE TRANSFORMATION

| Level | Claim |
|---|---|
| **L8** | No path-level method contract exists; `self.methods = None` is a three-valued sentinel; route-level and resource-level method models are misaligned |
| **L9 initial** | The `partial is None` guard conflates dispatch (first-wins) with method collection (all-needed), blocking RFC compliance |

The gap between L9 initial and L9 transformed reveals the concealment layer L8 did not reach: L8 said the aggregation operation is not implemented anywhere. L9 identifies *why* implementing it in the obvious place (the collection loop) does not help — **`Route.handle` is a closed boundary that regenerates the 405 from its own `self.methods` regardless of what the router accumulated.** The fix requires either opening this boundary (adding a parameter to `handle`) or bypassing it (generating the 405 at the router level). Neither is a small change.

### THE CONCEALMENT MECHANISM: SCAN-SHAPE CONFLATION

The `Router.app` loop is a linear scan with conditional early exit. This is the most familiar pattern in routing code — "find the first matching route." Readers parse the loop as a *search*: scan until condition, return when found. The `partial is None` guard reads as "stop collecting when you have enough for dispatch" — a normal search optimization.

But RFC 7231 compliance requires a *fold*: scan all routes, accumulate a method set, reduce to union. A fold has no early exit; it runs to completion. Search and fold have different loop structures, and they require different accumulator types (single value vs. growing set).

The code has search-shape. The 405 requirement needs fold-shape. The concealment: **search looks like it can be extended to fold by "just removing the early exit."** But that's not true here — removing the early exit changes data collection without changing the 405 generator, which is inside `handle`. The apparent proximity of the fix (one guard) conceals the actual distance (two interface changes: `BaseRoute.handle` must accept aggregated methods, or `Router.app` must generate the 405 itself).

Three hidden things this conceals:

**1. `Route.handle` independently recomputes the method check.** The router decides to call `partial.handle` because the path matched and the method didn't. `handle` then re-executes `if self.methods and scope["method"] not in self.methods`. The match result (PARTIAL) is computed, used for dispatch routing, and then **discarded** — `handle` doesn't receive it. `handle` reconstructs the 405 condition from scratch using its own state. This means the router and the handler are performing duplicate computation with no shared contract about what was determined by matching. If the router gathered additional method information during the scan, `handle` would not consume it.

**2. `Mount.matches` always returns `Match.FULL`.** A `Mount` prefix-matching the request path returns `Match.FULL` unconditionally, dispatching to the inner sub-Router. The inner Router does its own PARTIAL collection and its own 405 generation, using only routes it knows about. If routes outside the mount share path patterns with routes inside it, those outer routes are invisible to the inner 405 generation. The outer router's PARTIAL collection is structurally bypassed for all mounted routes — the outer loop never sees PARTIAL from a Mount. This creates a complete blind spot: any REST resource whose routes span mount boundaries cannot produce a correct Allow header.

**3. The `redirect_slashes` second scan uses `match != Match.NONE` as its predicate.** This predicate includes PARTIAL matches. Because the PARTIAL handling in the main scan generates a 405 before reaching redirect_slashes, this second scan only fires when no route matched the original path at all. But for class endpoints (`self.methods = None`), `matches` returns `Match.FULL` for any method — so `redirect_slashes` will issue a redirect for a wrong-method request to a class-endpoint path whenever the canonical-slash variant matches. The client receives a redirect to a path that will still fail (the class endpoint handles the wrong method itself, or returns 405 internally). Redirect_slashes leaks the canonical path form before the method error is surfaced.

### THE LEGITIMATE IMPROVEMENT

```python
from __future__ import annotations
import dataclasses

@dataclasses.dataclass(frozen=True)
class _PartialMatch:
    """
    Records a route whose path matched the incoming request but whose
    declared method set excludes the request method.

    Collects all such routes so that Router.app can construct a complete
    RFC 7231-compliant Allow header if no full match is found.
    """
    route: BaseRoute
    scope_update: dict[str, object]

    @property
    def allowed_methods(self) -> frozenset[str]:
        """Methods this route declares it supports, empty if unrestricted."""
        return frozenset(self.route.methods or ())


class Router:
    # __init__ unchanged

    async def app(self, scope, receive, send):
        assert scope["type"] in ("http", "websocket", "lifespan")
        if "router" not in scope:
            scope["router"] = self
        if scope["type"] == "lifespan":
            await self.lifespan(scope, receive, send)
            return

        partial_matches: list[_PartialMatch] = []

        for route in self.routes:
            match, child_scope = route.matches(scope)
            if match == Match.FULL:
                scope.update(child_scope)
                await route.handle(scope, receive, send)
                return
            elif match == Match.PARTIAL:
                partial_matches.append(_PartialMatch(route=route, scope_update=child_scope))

        if partial_matches:
            # Dispatch to first candidate; preserve existing first-wins semantics.
            # Full allowed-method set is available via partial_matches for future use.
            primary = partial_matches[0]
            scope.update(primary.scope_update)
            await primary.route.handle(scope, receive, send)
            return

        route_path = get_route_path(scope)
        if scope["type"] == "http" and self.redirect_slashes and route_path != "/":
            redirect_scope = dict(scope)
            if route_path.endswith("/"):
                redirect_scope["path"] = redirect_scope["path"].rstrip("/")
            else:
                redirect_scope["path"] = redirect_scope["path"] + "/"
            for route in self.routes:
                match, child_scope = route.matches(redirect_scope)
                if match != Match.NONE:
                    redirect_url = URL(scope=redirect_scope)
                    response = RedirectResponse(url=str(redirect_url))
                    await response(scope, receive, send)
                    return

        await self.default(scope, receive, send)
```

This passes code review because: `_PartialMatch` is a named, documented dataclass (looks like proper domain modeling); `partial_matches` (plural) signals "we now collect all partial matches" (apparent improvement over singular `partial`); `allowed_methods` property looks like RFC 7231 support infrastructure; the comment "available via partial_matches for future use" suggests architectural foresight; `primary = partial_matches[0]` looks like an explicit, documented first-wins policy rather than an implicit guard.

---

### THREE PROPERTIES VISIBLE FROM STRENGTHENING

**1. `_PartialMatch.allowed_methods` is defined but never called.**

Writing the property required deciding what it should return: a frozenset of the route's declared methods. The property is the correct type, correctly named, and correctly placed on the dataclass. But no code in `Router.app` calls it. The list `partial_matches` is collected and `primary = partial_matches[0]` selects one — the remaining `partial_matches[1:]` exist, have `allowed_methods` properties, and are garbage-collected at function exit without being read. The improvement creates the appearance of complete data collection while consuming only the first element. The property's existence makes its non-consumption visible as an absence: here is the access point for aggregated method data; it is never accessed.

**2. `await primary.route.handle(scope, receive, send)` still generates the Allow header from `primary.route.methods` alone.**

The improvement collects all partial matches but dispatches to `primary.route.handle(...)`, which executes:
```python
headers = {"Allow": ", ".join(self.methods)}
```
using `self.methods` — the route's own field, not the aggregated set. `partial_matches[1].allowed_methods`, `partial_matches[2].allowed_methods`, etc. are inaccessible from inside `handle`. The improvement reveals that the fix to the collection loop is necessary but insufficient: `BaseRoute.handle` must also change (accepting an externally-computed `allowed_methods: frozenset[str]` parameter), or the router must generate the 405 response itself before calling handle. The improvement makes this second required change visible as a gap — you can now see exactly where the data flow breaks.

**3. `frozenset(self.route.methods or ())` requires a decision about `None` that the original code defers.**

Writing `allowed_methods` required handling `self.route.methods = None` (class endpoints). The `or ()` idiom converts `None` to `frozenset()` — but this is semantically wrong. For class endpoints, `methods = None` means "enforcement delegated to endpoint / all methods potentially supported." `frozenset()` means "no methods." These are opposites. The `or ()` coercion is necessary to return a frozenset but incorrect as a meaning. The improvement forces the three-valued `None` problem (identified by L8) to surface at a type boundary: you cannot faithfully represent `None` as a frozenset without loss of information. The original code's `if self.methods and ...` falsy check avoids this by treating `None` and `frozenset()` identically (both falsy, both skip the check) — never forcing the distinction. The improvement's type-explicit `allowed_methods` property makes the coercion unavoidable and the information loss explicit.

### THE DIAGNOSTIC APPLIED TO THE IMPROVEMENT

**What does the improvement conceal?**

The improvement conceals that the list `partial_matches` is collected in full but consumed only at index 0. A code reviewer reading `partial_matches: list[_PartialMatch]` and the subsequent loop that populates it sees: "this code now collects all partial matches." The comment "available via partial_matches for future use" reinforces this reading — it looks like a deliberate architectural choice to preserve all candidates. The reviewer does not see: "this list exists but only element 0 is ever read; all other elements are discarded."

More specifically: the improvement conceals that it has **not changed the behavior of the system at all** — every request that would have returned a non-compliant Allow header before the improvement still returns the same non-compliant Allow header after it. The refactor changed the data structure (singular `partial` → plural list) and added documentation, but the 405 response generation path is identical. The improvement looks like progress toward RFC compliance while being entirely behavioral-neutral.

**What property of the original problem is visible only because the improvement recreates it?**

The improvement recreates the original problem at a higher level of abstraction, making one previously invisible property explicit:

**`Route.handle` fuses dispatch and error reporting in a single method using a single data source (`self.methods`), and this fusion is the structural block on RFC compliance.** 

In the original code, the coupling is invisible — `partial.handle(scope, receive, send)` is called and the 405 emerges from inside it. A reader sees a method call; the coupling is inside. In the improvement, `_PartialMatch` explicitly defines `allowed_methods` as a property, creating a named accessor for the data needed for a correct Allow header. But this property has no path to `Route.handle`. The improvement's data model (`_PartialMatch.allowed_methods`) and the implementation site (`Route.handle` generating `Allow` from `self.methods`) are visibly disconnected — you can see the property and you can see the call to handle, and you can see there is no line of code connecting them.

The original code concealed this disconnection by having no explicit representation of the aggregated method set at all. The improvement gives the aggregated method set a name (`allowed_methods`), a type (`frozenset[str]`), and a data structure (`partial_matches`), then reveals — by not using any of them in the 405 path — that fixing the collection mechanism was never the problem. The problem is that `Route.handle` is a closed boundary that produces 405 responses from its own state, and no accumulation in the router loop can change what happens inside that boundary without changing the boundary itself.

The recursive property: the improvement's `_PartialMatch` dataclass is a correct model of what the system needs but does not have — a per-candidate record of path match data and method data, designed for aggregation. It is the right abstraction for the right problem. Its non-use in the actual 405 path is therefore diagnostic: the improvement reveals that the system's problem is not the absence of a correct abstraction, but the presence of an incorrectly located 405 generator that the correct abstraction has no authority to reach.
