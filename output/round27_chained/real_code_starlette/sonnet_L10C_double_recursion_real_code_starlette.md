## Structure First: Level 10 Diagnostic

---

### THE INITIAL CLAIM

**`url_path_for` and `Router.app` are structurally independent algorithms that share the route list but not the resolution contract. `url_path_for` resolves by name → string substitution; forward routing resolves by path-regex priority ordering. Configurations exist where `url_path_for(name, **params)` generates URL P, but a forward request to P dispatches to a *different handler* than the named route — because string substitution knows nothing about regex priority. The round-trip invariant that the function name implies is neither documented, mechanically enforced, nor achievable with the current implementation.**

Falsifiable test:

```python
r = Router(routes=[
    Route("/api/special", special_handler, methods=["GET"], name="special"),
    Route("/api/{id}",    item_handler,    name="item"),
])
```

- `r.url_path_for("item", id="special")` → `Route("/api/{id}")` name matches → `replace_params` substitutes `"special"` for `{id}` → returns `"/api/special"`.
- `GET /api/special` → Router scans in registration order → `Route("/api/special")` regex `^/api/special$` matches first → `Match.FULL` → dispatches to `special_handler`.

Result: `url_path_for("item", id="special")` → `/api/special`, but `GET /api/special` → `special_handler`, not `item_handler`. The URL generated by the "item" route does not deliver the request to the "item" route.

---

### THE THREE-EXPERT DIALECTIC

**Expert 1 — Defender:** The test is clean and the mechanism is exact. String substitution (`replace_params` into `path_format`) is path-local: it produces the string that the route's own pattern would generate, with no awareness that an earlier route with a literal segment claims the same string. The two algorithms — name-first exception-propagation and regex-first linear scan — are not inverse functions of each other. They share input (the route list) but compute on different criteria with different short-circuit logic. No code in the system connects them. The round-trip failure is therefore structural, not accidental.

**Expert 2 — Attacker:** This is a configuration hazard, not a code defect. Every routing system with priority ordering and parametric paths has this property — literal routes shadow parametric ones. The claim conflates two different contracts: `url_path_for` is documented as "generate the URL path *for this named route*" (a per-route URL computation), not "generate a URL *guaranteed to dispatch to this route*" (a system-level inverse). The Starlette documentation makes no round-trip promise. The fix is: don't register parametric routes that can collide with literals, or enforce naming discipline. The code does what the documentation says.

**Expert 3 — Probing:** You're both arguing about whether `url_path_for` should provide round-trip fidelity. Neither of you has examined what method should appear in `url_path_for`'s signature if it were to provide that fidelity. Consider: `Route("/data", get_handler, methods=["GET"], name="data")` and `Route("/data", post_handler, methods=["POST"], name="data")`. Call `url_path_for("data")` — you get `/data`. You're building a form whose action attribute will POST. The URL is correct structurally but wrong methodologically: the "data" url resolved from the GET route, which would 405 on POST. The round-trip problem is not just "wrong handler for this URL" — it's "wrong method awareness entirely." And the reason `url_path_for` has no method parameter is that URL paths in REST are resource identifiers, not operation identifiers. `url_path_for` is one-dimensional (URL only); forward routing is two-dimensional (URL × method → handler). Is the real question whether `url_path_for` has a bug, or whether the routing system is trying to invert a 2D mapping with 1D information?

---

### THE TRANSFORMATION

| Level | Claim |
|---|---|
| **L8** | `self.methods = None` is a three-valued sentinel; route-level and resource-level method models are misaligned |
| **L9** | 405 generation is inside `Route.handle` (wrong level) and `Route.handle` is a closed boundary that independently regenerates the method check, making collection-loop fixes insufficient |
| **L10 initial** | `url_path_for` and forward routing are structurally independent; configurations exist where the generated URL does not deliver to the generating route |
| **L10 transformed** | `url_path_for` is a projection of a two-dimensional routing system (URL × method → handler) onto a one-dimensional space (name → URL). The dropped dimension — method — is precisely the dimension where L8 identified a three-valued sentinel and L9 identified a closed boundary. Every mechanism that needs complete method information hits the same boundary: class endpoints hold their method information internally (inside the class's handler dispatch), not in routing metadata. The one-dimensional `url_path_for` is not a bug; it reflects the system's structural choice to split method dispatch across two levels — routing layer (function endpoints: `self.methods`) and endpoint layer (class endpoints: internal dispatch) — with `self.methods = None` as the interface between them. |

The gap between L10 initial and L10 transformed: the initial claim locates the failure in the `url_path_for` algorithm. The transformed claim locates it in the **dimensional split** — the routing system models method dispatch at two levels with no shared interface, and the missing method dimension in `url_path_for` is the same incompleteness that produces the Allow header error (L9) and the three-valued sentinel (L8). These are not three separate bugs; they are three surface manifestations of one structural property.

### THE CONCEALMENT MECHANISM: INVERSION ILLUSION

`url_path_for(name, /, **path_params)` has a signature that mirrors forward routing's parameter extraction: forward routing captures `path_params` from the URL regex; `url_path_for` accepts `path_params` to substitute back in. The symmetry of the parameter interface creates the impression of algorithmic inversion — "these are the parameters I extracted going forward; I provide them to get the URL back."

But the inputs to forward routing are `(path, method, scope)`. The inputs to `url_path_for` are `(name, path_params)`. Two dimensions are absent from the reverse: `method` and `scope`. The method is missing because URL paths are resource identifiers. The scope is missing because per-request context (auth, headers) is not part of route identity. `url_path_for` appears to be a complete inverse because the one shared dimension (path parameters) is symmetric — but it's a projection, not an inversion.

The concealment: **parameter symmetry makes `url_path_for` look like the mathematical inverse of forward routing while concealing that it operates on a strictly smaller information set.** The missing dimension (method) is exactly the dimension that carries all the correctness problems visible in L8 and L9.

Three things this concealment hides:

**1. `Route.url_path_for` has no access to sibling routes and cannot detect shadowing.** `url_path_for` is a method on `BaseRoute` — each route resolves its own URL by examining `self.name`, `self.path_format`, and `self.param_convertors`. Shadowing is a property of the full route list (does any earlier route with a more specific pattern claim this URL?). The per-route method encapsulation feels natural — each route knows its own pattern — but correctness requires the router-level view. `Route.url_path_for` produces a URL that would be correct in isolation, with no mechanism to detect that it will be intercepted by an earlier route in dispatch.

**2. The exception-propagation control flow in `Router.url_path_for` has the same visual shape as the `Match` dispatch loop but resolves by different criteria.** The `try: return route.url_path_for(name, **path_params) / except NoMatchFound: pass` pattern looks like a mirror of the `if match == Match.FULL: return / elif match == Match.PARTIAL: ...` pattern. Both are linear scans with early exit. But the exit conditions are different: forward routing exits on regex match priority; reverse routing exits on name match priority. A route that is second by regex (shadowed by a literal pattern) can be first by name (no other route has that name). These orderings are independent, and the visual symmetry conceals their independence.

**3. `url_path_for` does not verify that the generated URL is reachable for any particular method.** A route registered with `methods=["POST"]` produces a URL via `url_path_for`. If the caller uses that URL in a GET link, they get a 405. This is obvious once stated — but `url_path_for`'s method-free signature means the question "is this URL valid for my intended method?" never arises at the call site. The absence of a method parameter conceals that method is a required input for answering "can this URL be used?"

### THE LEGITIMATE IMPROVEMENT

```python
import dataclasses
from typing import Optional

@dataclasses.dataclass(frozen=True)
class RouteURL:
    """
    A URL generated by url_path_for, annotated with the source route's
    method contract for caller use.

    Allows callers to determine at link-generation time whether the URL
    supports their intended HTTP method.

    Attributes
    ----------
    path     : The generated URL path string.
    protocol : "http" or "websocket".
    name     : The route name that generated this URL.
    methods  : Methods the source route declares it supports, or None if the
               source route is a class-based endpoint (methods determined
               internally by the endpoint class, not the routing layer).
    """
    path: str
    protocol: str = "http"
    name: str = ""
    methods: Optional[frozenset[str]] = None

    def __str__(self) -> str:
        return self.path

    def supports(self, method: str) -> Optional[bool]:
        """
        Return True if this route declares support for *method*, False if it
        declares that it does not, or None if the route is a class-based
        endpoint whose method support is not declared at the routing layer.

        Callers should treat None as "unknown — check endpoint documentation"
        rather than as True or False.
        """
        if self.methods is None:
            return None   # class endpoint: methods determined by endpoint, not router
        return method.upper() in self.methods


class Route(BaseRoute):
    # __init__ unchanged

    def url_path_for(self, name: str, /, **path_params) -> RouteURL:
        seen_params = set(path_params.keys())
        expected_params = set(self.param_convertors.keys())
        if name != self.name or seen_params != expected_params:
            raise NoMatchFound(name, path_params)
        path, remaining_params = replace_params(
            self.path_format,
            self.param_convertors,
            dict(path_params),           # copy: don't mutate caller's dict
        )
        assert not remaining_params
        return RouteURL(
            path=path,
            protocol="http",
            name=self.name,
            methods=frozenset(self.methods) if self.methods is not None else None,
        )
```

This passes code review because: `RouteURL` is a clean, documented dataclass (looks like proper domain modeling); the `supports()` method with `Optional[bool]` return type looks like careful, principled handling of the class-endpoint `None` case ("unknown" is better than a wrong answer); the docstring on `methods` explicitly acknowledges the routing-layer / class-endpoint split, which reads as architectural sophistication; `dict(path_params)` copy reads as a correctness fix for `replace_params`' mutation of its argument; `frozenset` conversion reads as appropriate immutability for a frozen dataclass.

---

### THREE PROPERTIES VISIBLE BECAUSE OF THE IMPROVEMENT

**1. `supports()` returns `None` for class endpoints — the three-valued incompleteness (L8) propagates out of the routing layer into every caller.**

Writing `supports()` required choosing a return value for `self.methods = None`. The only honest choice is `None` — "unknown at this layer." But this means every caller of `supports()` receives a value that means neither True nor False. Code that writes `if url.supports("DELETE"):` silently treats `None` as falsy. Code that writes `if url.supports("DELETE") is not False:` silently treats `None` as permissive. The three-valued sentinel that L8 identified inside `Route.matches` (`if self.methods and ...`) has now been surfaced to the caller's `if` statement. The improvement correctly represents the incompleteness — and makes visible that there is no resolution at the routing layer. The `None` can only be propagated or suppressed, not answered.

**2. `RouteURL.methods` is per-route, not per-resource — two `url_path_for` calls at the same path return conflicting method information.**

Writing the `methods` field required deciding whose methods to report. The only available data is `self.methods` of the generating route. But for a resource with both a GET-only and POST-only route at the same path:

```python
r = Router(routes=[
    Route("/data", get_handler, methods=["GET"], name="get_data"),
    Route("/data", post_handler, methods=["POST"], name="post_data"),
])
url_get = r.url_path_for("get_data")   # RouteURL(path="/data", methods={"GET","HEAD"})
url_post = r.url_path_for("post_data")  # RouteURL(path="/data", methods={"POST"})
url_get.supports("POST")   # False — but POST /data works fine
url_post.supports("GET")   # False — but GET /data works fine
```

The two `RouteURL` objects have the same path but contradictory method information. The correct `methods` for either is `{"GET", "HEAD", "POST"}` — the union over all routes at this path. But `Route.url_path_for` has no access to sibling routes. Writing the improvement makes visible that the correct `methods` field requires Router-level information, but `url_path_for` is a `BaseRoute` method. The improvement recreates the per-route / per-resource gap that L9 identified in the Allow header — now on the reverse-routing side.

**3. `RouteURL` is not `str`, and this breaks the existing `URLPath(str)` inheritance contract.**

`URLPath` inherits from `str`. This is not incidental — it allows generated URLs to be used wherever strings are expected: `RedirectResponse(url=url_path_for("home"))`, f-string interpolation, `URL(scope={"path": url_path_for("route")})`. Writing `RouteURL` as a dataclass (the natural choice for a structured return type) breaks all these callsites. The improvement reveals that `str` inheritance in `URLPath` is a **callsite compatibility constraint**, not a convenience. The dataclass form is structurally better (typed fields, documented attributes, immutable) but API-incompatible. Any improvement that adds method metadata to the URL object must choose between structural correctness (dataclass) and callsite compatibility (str subclass), and neither choice is free.

### DIAGNOSTIC APPLIED TO THE IMPROVEMENT

**What does the improvement conceal?**

`RouteURL.supports("POST")` looks like it answers "can I POST to this URL?" The question feels answered — you got a structured object with a typed field and a method with a clear docstring. The concealment: `supports()` answers "does the *route that generated this URL* declare POST support?" — not "does a POST request to this URL succeed?" For two routes at the same path, `supports("POST")` on the GET route's URL returns `False`, but POST to the URL succeeds (dispatched to the POST route). The improvement creates a named, typed, documented interface that gives a wrong answer for multi-route resources. The wrong answer is more dangerous than no answer, because it will not generate `None` (which callers might check for) — it generates `False` (which callers will act on).

**What property of the original problem is visible only because the improvement recreates it?**

The improvement forces the decision: whose `methods` go in `RouteURL.methods`? The only available answer is the generating route's. This answer is correct for the individual route and incorrect for the resource. The improvement thereby demonstrates: **`url_path_for` is structurally a per-route operation, but method correctness is a per-resource property.** The original code never faces this tension because `URLPath` carries no method information — the question "which route's methods?" never arises. The improvement, by adding a `methods` field, makes the per-route / per-resource gap unavoidable and names it.

This is the same gap L9 identified on the forward-routing side — `Route.handle` generates the Allow header from `self.methods` (per-route) when the correct Allow header requires the union across all path-matching routes (per-resource). The improvement recreates that gap in the reverse-routing direction: `url_path_for` populates `RouteURL.methods` from the generating route (per-route) when the correct methods field requires the union across all routes at the path (per-resource). The gap is the same structural property, appearing symmetrically in both routing directions.

### THE SECOND IMPROVEMENT

Addresses the per-route / per-resource gap by moving method aggregation to the Router level, where sibling routes are visible:

```python
class Router:
    def url_path_for(self, name: str, /, **path_params) -> RouteURL:
        """
        Generate the URL path for the named route with the given parameters,
        annotated with the complete method set for the resource at that path.

        Unlike per-route url_path_for, this Router-level method aggregates
        methods across all routes that match the generated path, so that
        RouteURL.methods reflects the full resource contract rather than
        only the generating route's declared methods.
        """
        # Phase 1: find the named route and generate the URL
        primary_url: Optional[RouteURL] = None
        for route in self.routes:
            try:
                primary_url = route.url_path_for(name, **path_params)
                break
            except NoMatchFound:
                pass

        if primary_url is None:
            raise NoMatchFound(name, path_params)

        # Phase 2: scan all routes at the generated path to build the
        # complete resource-level method set.
        probe_scope = {
            "type": "http",
            "method": "GET",        # sentinel: PARTIAL != NONE for wrong-method routes
            "path": str(primary_url),
            "root_path": "",
            "query_string": b"",
            "path_params": {},
        }
        aggregated_methods: set[str] = set()
        has_unrestricted = False   # True if any matched route has methods=None

        for route in self.routes:
            match, _ = route.matches(probe_scope)
            if match != Match.NONE:
                route_methods = getattr(route, "methods", None)
                if route_methods is None:
                    has_unrestricted = True
                else:
                    aggregated_methods.update(route_methods)

        return RouteURL(
            path=str(primary_url),
            protocol=primary_url.protocol,
            name=primary_url.name,
            methods=None if has_unrestricted else frozenset(aggregated_methods),
        )
```

---

### DIAGNOSTIC APPLIED TO THE SECOND IMPROVEMENT

**What does the improvement conceal?**

The probe scope uses `"method": "GET"` as a sentinel. The comment explains the logic: `Match.PARTIAL != Match.NONE`, so wrong-method routes are detected. But Mount routes return `Match.FULL` for any request to any path under their prefix — including GET. When the probe path falls under a Mount, the Mount matches (`Match.FULL`), is counted as a "matched route," but `getattr(route, "methods", None)` on the Mount returns `None` (Mounts have no `methods` attribute) → `has_unrestricted = True` → `RouteURL.methods = None`.

The concealment: the Mount's `None` is a false `None`. Mount's `methods = None` (attribute absent, defaulting to `None`) means "we didn't declare methods" — which is different from `Route.methods = None` which means "class endpoint, methods unknown." The improvement cannot distinguish these two reasons for `None`: the Mount's absence of a `methods` attribute is conflated with the class-endpoint's deliberate `None` sentinel. A resource with one GET-only route inside a Mount will report `methods = None` ("unknown") instead of `methods = {"GET", "HEAD"}` ("known, GET only"), because the Mount itself counts as an unrestricted route. The aggregation is wrong for any mounted route.

**What property of the original problem is visible only because the improvement recreates it?**

The probe scan's `for route in self.routes` iterates the Router's direct routes — which includes Mounts as opaque entries. The inner routes of a Mount (the routes that actually have method declarations) are invisible from `self.routes`. To reach them, the probe would need to enter the Mount's sub-Router, which requires traversing the route tree recursively.

The second improvement recreates L9's second hidden property: "If routes outside the mount share path patterns with routes inside it, those outer routes are invisible to the inner 405 generation. The outer router's PARTIAL collection is structurally bypassed for all mounted routes." Here, the *outer router's method aggregation* is structurally incomplete for mounted routes — the inner routes' method declarations are invisible from `self.routes`. The same opacity appears in both the forward-routing (L9) and reverse-routing (L10) aggregation problems, because both require traversing the full route tree, and the Mount's internal structure is opaque to iteration at the Router level.

---

### THE STRUCTURAL INVARIANT

The property that persists through every improvement — from L8's sentinel, through L9's closed boundary, through L10's method aggregation — because it is a property of the problem space, not the implementation:

**Method dispatch in this routing system exists at two levels simultaneously: at the routing level for function endpoints (`self.methods`, checked in `Route.matches` and `Route.handle`) and at the endpoint level for class endpoints (internal method dispatch inside the class, invisible to the routing layer). The interface between these levels is `self.methods = None`: this sentinel means "method enforcement is delegated downward to the endpoint; the routing layer has no information." Every mechanism that requires complete method information — 405 Allow header construction (L9), `url_path_for` method annotation (L10), per-resource method aggregation (L10's second improvement) — will be correct for function endpoints and structurally incomplete for class endpoints, because completeness requires information that the routing layer has explicitly chosen not to hold. No improvement to the routing layer's collection logic, data structures, or aggregation algorithm can eliminate this incompleteness without either (1) requiring class endpoints to register their method support at route construction time, collapsing the two-level model into one, or (2) querying the endpoint class at request time before dispatch, coupling routing to endpoint implementation. The `None` sentinel is not a bug — it is the contract boundary between two architectural levels. Every improvement recreates the boundary at a new location because the boundary is a property of the architecture, not of any particular code site.**
