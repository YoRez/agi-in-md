# Structural Analysis: `AdaptiveRateGovernor`

---

## I. The Claim

**The feedback loop is not closed — it operates on phantom correlation.**

`allow()` tracks request counts. `record_outcome()` tracks quality signals. `_maybe_adjust()` reads both and modifies `_rate_limit`. But no mechanism links a specific request to its outcome. The governor adjusts throughput as if quality is a function of rate, while possessing zero evidence that any causal relationship exists between the two streams.

**Falsifiable test**: In a system where high request rate causes quality degradation with 30-second latency, the governor will *increase* `_rate_limit` during the degradation period — because it reads quality from `[T-60, T]`, which reflects the low-rate past, while approving requests in a high-rate present.

---

## II. Three Experts

**Expert A (defends):**
The misattribution is structural. When `_maybe_adjust` fires, the outcomes it reads were generated by requests that may have expired from `_requests` 30 seconds ago. The controller is adjusting a knob based on readings from a sensor measuring a different moment. This isn't noise — it's systematically backwards during any transient, which is precisely when correct adaptation matters.

**Expert B (attacks):**
This is a feature misread as a bug. Aggregate feedback loops don't require request-outcome coupling. CPU governors don't track which instruction caused thermal throttling. The window average is a deliberate design for noisy environments. The claim mistakes a simplification appropriate for steady-state signals as a defect.

**Expert C (probes what both assume):**
Both of you assume the feedback loop *is what it claims to be*. Consider: `_rate_limit` starts as `int(10)`, grows as `float(14.7)` after `* (1 + 0.5)`, and is compared against `len(self._requests)` — an integer. The code works by accident of Python's comparison semantics. More critically: is `_rate_limit = 10` ten requests per *window* or ten requests per *second*? `_expire` creates a sliding 60-second window. The code never normalizes. You're debating a feedback loop whose **units are undefined**.

### The Transformation

**Original claim:** The feedback loop operates on phantom correlation — requests and outcomes aren't linked.

**Transformed claim:** The feedback loop's units are undefined and internally inconsistent. `_rate_limit` is simultaneously: (a) a count threshold compared against integer window occupancy, (b) a float growing multiplicatively without normalization, (c) an implicit rate whose denomination (per-second? per-window?) is never established. Even if causality were correct, the feedback would be dimensionally incoherent.

**The gap:** The original claim identified a *causality problem*. The transformed claim reveals a prior *measurement problem* — the system doesn't know what it's measuring, so causal correctness would be insufficient.

---

## III. The Concealment Mechanism

**Plausible coherence through naming.**

`AdaptiveRateGovernor`, `allow()`, `record_outcome()`, `get_pressure()`, `_sensitivity` — this vocabulary implies a closed feedback control loop. Each method, read individually, looks reasonable. The problems exist only in the *relationships between methods*:

- `allow()` uses a float as a count threshold → type incoherence invisible at method boundary
- `record_outcome()` has no knowledge of which request it responds to → causal gap invisible at call site
- `get_pressure()` divides by the same float that `allow()` uses as an integer → two callers of the same variable see different semantics
- `_adjustments` logs `new_rate` but not the window-relative `_rate_limit` that was in effect *when the logged outcomes were generated* → audit trail is causally incomplete

The concealment works because the code is readable at the line level and broken at the system level.

---

## IV. Improvement One: Legitimate, Concealment-Deepening

```python
def allow(self):
    now = time.time()
    self._expire(now)
    effective_limit = int(self._rate_limit)  # normalize float to integer for comparison
    if len(self._requests) >= effective_limit:
        return False
    self._requests.append(now)
    return True
```

**Why it passes review:** Fixes an obvious type inconsistency. Float-to-int normalization at point of use is standard practice. Reviewers see "corrected comparison semantics."

**Why it deepens concealment:**
1. The float-vs-int surface symptom disappears, removing the visual cue that `_rate_limit` has a type identity problem
2. `_rate_limit` now has two representations: float in storage, int at enforcement — this *looks* like intentional design (smooth accumulation, discrete enforcement)
3. `get_effective_rate()` now returns a float while `allow()` enforces an int: two methods describe the same rate with different values, silently

---

## V. Three Properties Visible Only Because We Strengthened It

**1. `_rate_limit` is architecturally homeless.**
When we added `int()`, we had to *choose where to canonicalize*. `allow()`? `get_effective_rate()`? `_maybe_adjust()`? There is no correct location because the variable serves three masters with incompatible type requirements. The fix forced the question; the question has no answer.

**2. The audit trail records a phantom quantity.**
`_adjustments` logs `new_rate`. After our fix, `new_rate` is a float, but enforcement uses `int(new_rate)`. The log records a value the system never enforced. This was always true — the fix made the divergence explicit.

**3. `get_pressure()` is broken in a newly visible way.**
```python
return len(self._requests) / self._rate_limit  # float denominator
```
After the fix, `allow()` uses `int(self._rate_limit)` as its ceiling, but `get_pressure()` uses the raw float. Pressure > 1.0 becomes possible *by design* — a full queue by the integer ceiling reads as fractionally over-pressured by the float denominator. We made the incoherence operational.

## VI. Improvement Two: Contradicts the First

```python
def _maybe_adjust(self, now):
    if len(self._outcomes) < 5:
        return
    recent = [s for _, s in self._outcomes]
    avg_quality = sum(recent) / len(recent)
    if avg_quality > 0.8:
        new_rate = self._rate_limit * (1 + self._sensitivity)
    elif avg_quality < 0.3:
        new_rate = self._rate_limit * (1 - self._sensitivity)
    else:
        new_rate = self._rate_limit
    # Store rate_limit as integer throughout — it is fundamentally a count
    self._rate_limit = max(1, min(round(new_rate), 1000))
    self._adjustments.append({
        "time": now, "new_rate": self._rate_limit,
        "quality": avg_quality, "samples": len(recent)
    })
```

**Why it passes review:** Rate limits are counts; storing them as integers is semantically correct. `round()` at write time is standard for discrete-valued accumulation. Reviewers see "enforced integer semantics at the authoritative assignment."

**The contradiction:** Improvement 1 says *store float, convert at use*. Improvement 2 says *store integer, round at write*. Both are defensible. They are opposite answers to the same question.

---

## VII. The Structural Conflict

**The adaptive algorithm and the rate enforcement mechanism require different types for the same variable.**

The algorithm wants a float: multiplicative updates of `10 * 1.5 = 15 → 15 * 1.5 = 22.5` require fractional precision to be stable and path-dependent. Near the floor, `1 * 0.5 = 0.5`, which rounds to 1 — the floor becomes a trap.

The enforcer wants an integer: it is compared against `len()`, reported as a capacity, divided into a count for pressure — all semantically discrete operations.

**This conflict is only visible because both improvements are legitimate.** If only one had been proposed, the type issue would appear "solved." The existence of two correct, contradictory solutions proves the variable is trying to be two things.

---

## VIII. Improvement Three: Resolves the Conflict

```python
class AdaptiveRateGovernor:
    def __init__(self, initial_rate=10, window=60, sensitivity=0.5):
        self._target_rate = float(initial_rate)   # continuous: for adaptation
        self._effective_rate = int(initial_rate)  # discrete: for enforcement
        self._window = window
        self._sensitivity = sensitivity
        self._requests = deque()
        self._outcomes = deque()
        self._adjustments = []

    def allow(self):
        now = time.time()
        self._expire(now)
        if len(self._requests) >= self._effective_rate:
            return False
        self._requests.append(now)
        return True

    def _maybe_adjust(self, now):
        if len(self._outcomes) < 5:
            return
        recent = [s for _, s in self._outcomes]
        avg_quality = sum(recent) / len(recent)
        if avg_quality > 0.8:
            self._target_rate = min(self._target_rate * (1 + self._sensitivity), 1000)
        elif avg_quality < 0.3:
            self._target_rate = max(self._target_rate * (1 - self._sensitivity), 1)
        self._effective_rate = round(self._target_rate)
        self._adjustments.append({
            "time": now,
            "target_rate": self._target_rate,
            "effective_rate": self._effective_rate,
            "quality": avg_quality,
        })

    def get_effective_rate(self):
        return self._effective_rate

    def get_pressure(self):
        if self._effective_rate == 0:
            return 1.0
        return len(self._requests) / self._effective_rate
```

**How it fails:**

The split creates a *quantization dead zone*. `_target_rate` can oscillate between `7.5` and `8.25` while `_effective_rate` stays pinned at `8`. The governor *appears stable* during active oscillation — the audit log shows a constant `effective_rate: 8` while `target_rate` yo-yos. The observable quantity is frozen; the internal state is unstable. Worse: near the floor at `_target_rate = 1.0`, a decrease yields `max(1.0 * 0.5, 1) = 1.0`, an increase yields `1.0 * 1.5 = 1.5 → effective_rate = 2`. The adjustment is **asymmetric**: the system can increase but not decrease below 1. The floor creates a ratchet the code doesn't know it has.

**What the failure reveals about the design space:**

The quantization dead zone exposes a constraint the conflict alone could not: **the feedback loop's minimum meaningful signal is bounded below by the integer step size at low rates, but the sensitivity parameter implies sub-integer precision that can never be enforced.**

At `_rate_limit = 2` with `sensitivity = 0.5`, the theoretical decrease is to `1.0` — one step. But at `_rate_limit = 100`, the theoretical decrease is to `50` — fifty steps of capacity removed in one adjustment. The adaptation velocity is *rate-dependent* in a way the code's flat sensitivity parameter doesn't model. The split variable makes this visible because now `_target_rate` can show you the continuous computation while `_effective_rate` shows you the discrete consequence — and the gap between them grows as rate grows.

## IX. The Redesign: Accepting the Topology

```python
class AdaptiveRateGovernor:
    """
    A burst-budget governor with observational adaptation.
    
    Enforces a sliding-window burst ceiling (integer) that is
    incrementally adjusted based on quality signals. Makes no
    claim about causality between rate and quality — treats
    quality as an environmental signal, not a rate function.
    Acknowledges that outcomes may arrive after their causing
    requests have expired.
    """
    def __init__(self, initial_budget=10, window=60, quality_window=None):
        self._window = window
        self._quality_window = quality_window or (window * 2)
        self._burst_budget = max(1, int(initial_budget))
        self._requests = deque()
        self._outcomes = deque()
        self._adjustments = []

    def allow(self):
        now = time.time()
        self._expire_requests(now)
        if len(self._requests) >= self._burst_budget:
            return False
        self._requests.append(now)
        return True

    def record_outcome(self, quality_score):
        now = time.time()
        self._outcomes.append((now, quality_score))
        self._expire_outcomes(now)
        self._maybe_adjust(now)

    def _expire_requests(self, now):
        cutoff = now - self._window
        while self._requests and self._requests[0] < cutoff:
            self._requests.popleft()

    def _expire_outcomes(self, now):
        # Outcomes expire on their own longer schedule —
        # acknowledged that they arrive after their requests
        cutoff = now - self._quality_window
        while self._outcomes and self._outcomes[0][0] < cutoff:
            self._outcomes.popleft()

    def _maybe_adjust(self, now):
        if len(self._outcomes) < 5:
            return
        scores = [s for _, s in self._outcomes]
        avg_quality = sum(scores) / len(scores)
        prev_budget = self._burst_budget
        if avg_quality > 0.8:
            self._burst_budget = min(self._burst_budget + 1, 1000)
        elif avg_quality < 0.3:
            self._burst_budget = max(self._burst_budget - 1, 1)
        if self._burst_budget != prev_budget:
            self._adjustments.append({
                "time": now,
                "burst_budget": self._burst_budget,
                "quality": avg_quality,
                "sample_window": self._quality_window,
                "sample_count": len(scores),
            })

    def get_effective_rate(self):
        return self._burst_budget

    def get_pressure(self):
        if self._burst_budget == 0:
            return 1.0
        return len(self._requests) / self._burst_budget
```

**What it sacrifices:**

1. **Adjustment velocity at scale.** Replacing multiplicative adjustment with `±1` means: starting from 10, reaching 100 takes 90 quality-high events; the original reached it in `⌈log₁.₅(100/10)⌉ = 6`. At high budgets, additive steps are nearly inert.

2. **`_sensitivity` as a meaningful parameter.** It is removed entirely. Callers who expected to tune adaptation speed have no lever. This is a breaking API change.

3. **The claim to be an adaptive controller.** The redesign is explicitly *observational*: it reacts to quality without asserting causality. It sacrifices the narrative that it *manages* quality in exchange for the honesty that it *responds* to it.

**What the redesign reveals was never a flaw:**

The `_sensitivity` parameter was not a bug. It was the cost of attempting something impossible: **continuously variable, causally-grounded adaptive rate control** in a system where enforcement must be discrete, outcomes arrive asynchronously, and the causal link between rate and quality is unverifiable.

`_sensitivity` was the single scalar trying to simultaneously:
- Smooth the continuous-to-discrete transition
- Model the causal transfer function between rate and quality
- Bridge the temporal gap between request and outcome

Its failure modes — float/int mismatch, phantom correlation, asymmetric floor behavior — are not accidents of careless coding. They are the *necessary consequences* of encoding an impossible requirement as a single float parameter.

The redesign reveals that the original's real flaw was *ambition about what it was*: it described itself as a causal adaptive controller while possessing only the machinery of an observational one. `_sensitivity` is the scar tissue of that description. The `±1` replacement isn't better engineering — it's the honest acknowledgment that the design space doesn't contain a causally-grounded adaptive rate controller for asynchronous quality signals. It contains only governors that watch and react, and governors that pretend to do more.
